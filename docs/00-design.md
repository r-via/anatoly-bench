# anatoly-bench вЂ” Design document

> Archive of the design discussion for `anatoly-bench`, a benchmarking suite for the [Anatoly](https://github.com/r-via/anatoly) deep-audit agent.

**Status:** design agreed, implementation not yet started.
**Primary author of Anatoly:** RГ©mi Viau
**Date of this archive:** 2026-04-23

## 1. Goal

Build a standalone project вЂ” **`anatoly-bench`** вЂ” that measures Anatoly's performance at detecting code issues across its seven evaluation axes (correction, overengineering, utility, duplication, tests, best practices, documentation).

The bench must be able to:

1. Ingest a complete Anatoly audit report (sharded Markdown + metrics).
2. Compare the findings against a curated ground-truth catalog of known defects planted in a reference project.
3. Produce a quantitative score вЂ” per axis and global вЂ” that reflects both **recall** (did Anatoly find the planted defects?) and **precision** (did Anatoly flag things that were not defects?).

The primary long-term use is **regression detection**: when a prompt, a model, or a pipeline component in Anatoly changes, the bench should immediately surface whether the change improved or degraded detection quality.

## 2. Key design decisions (and their rationale)

### 2.1. Anatoly and anatoly-bench do not know each other

The bench consumes only **public artifacts** produced by Anatoly (the sharded report, `run-metrics.json`) via subprocess CLI calls. It does not import Anatoly's TypeScript types, does not share schemas, does not live in the same repo.

**Rationale:** this forces Anatoly's report format to be a public contract. If Anatoly changes the report format, the bench breaks вЂ” which is the correct signal: a breaking change happened and needs documentation. It also allows the bench to be versioned, shipped, and maintained independently.

### 2.2. Reference project generated by Evolve

Rather than hand-writing the reference codebase with planted defects, we use [Evolve](https://github.com/r-via/evolve) вЂ” another tool in the same ecosystem вЂ” to generate it from a detailed specification. Evolve reads a spec file (via its `--spec` flag, pointing to an arbitrary path) and converges a project iteratively until a `--check` script passes.

**Rationale:** the SPEC becomes a single source of truth that is both human-readable (for us and for Evolve) and machine-parseable (for the bench scorer). Regenerating the fixture after a spec change is a one-command operation. Because `--spec` accepts any path, the spec stays physically outside the directory Anatoly audits вЂ” no file-swap dance during generation.

### 2.3. One realistic project, all axes mixed

Instead of 70 isolated mini-fixtures (10 per axis Г— 7 axes), we use **a single realistic reference project of ~2000вЂ“3000 lines of code, with ~30 defects planted across all axes simultaneously**.

**Rationale:** Anatoly is designed to triage signal within real codebases. An isolated 20-line fixture with a single bug measures its ability to diagnose a highlighted case вЂ” not its ability to do its actual job, which is distinguishing real issues from noise. A multi-axis project with realistic ambient code (neutral logic, plausible abstractions, domain-appropriate idioms) is both:
- **Representative** of how users will actually invoke Anatoly (`anatoly run` on a real backend)
- **Honest** about precision: ambient code creates opportunities for false positives that isolated fixtures cannot reproduce

### 2.5. The reference project: slot-engine

A **headless TypeScript library + CLI for a video slot machine**. The project simulates reels, paylines, paytables, wild/scatter mechanics, free spins, and payout computation, with a built-in Monte Carlo simulator for measuring long-run RTP (return-to-player), hit frequency, and variance.

**Why this domain:** a slot machine has **business invariants** that are not mere technical correctness rules. Code can be perfectly type-checked, compile cleanly, and still be a catastrophic slot machine (RTP above 100%, wild substitutes for jackpot, free-spins that re-trigger indefinitely, etc.). This creates a uniquely rich test for Anatoly because it forces the agent to reason about **domain knowledge**, not just technical correctness.

**The specific framing (critical):** the initial implementation produced by Evolve is **anomalously favorable to the player** (RTP > 100%). The "bugs" are therefore a mix of:

- **Business-invariant violations**: code that works technically but breaks the economic invariants of a slot machine (uniform reel weights, unbounded free-spin chains, wild substituting for jackpot, paytable caps ignored, etc.)
- **Technical defects**: classical bugs (off-by-one in payline evaluation, `Math.random()` as RNG, `parseInt` without radix, unvalidated stdin, etc.)
- **Architectural defects**: dead modules, structural duplicates, over-engineered validator factories, missing tests, missing JSDoc.

Anatoly's job is to detect both categories. A competent auditor should, in effect, "restore the casino's edge" by flagging the code that leaks money to the player.

**Ethical framing:** the fixture is pure simulation (no real money, no UI, no gambling product). We deliberately avoid planting dark-pattern mechanics (near-miss psychology, post-loss compensation inversion, etc.). One borderline mechanic вЂ” a "session luck adjustment" that biases the RNG after player losses вЂ” is included specifically because it violates INV-MEMORY (the memorylessness invariant) and is forbidden in regulated markets. It serves as a hard test case for invariant-aware auditing.

**Noise:** ~70% of the generated code is ordinary neutral logic (CLI plumbing, formatters, RNG abstractions, simulation runner, configuration loading). This noise is essential: without it, precision is meaningless because there is nothing for Anatoly to confuse as a defect.

### 2.6. The SPEC.md document

`catalog/slot-engine/SPEC.md` is the single source of truth. It contains:

1. **Critical instructions for Evolve** вЂ” insisting that the defects must be preserved, must not be annotated as bugs, and must not be accompanied by additional defects.
2. **Domain overview** вЂ” enough context about slot machines for Evolve to generate realistic code.
3. **Functional spec** вЂ” API surface, project layout, file list (26 files, enforced).
4. **Business invariants (INV-RTP, INV-WEIGHTS, INV-WILD, INV-BET, INV-CAP, INV-MEMORY, INV-FREESPIN, INV-RNG)** вЂ” the economic correctness rules the engine must eventually violate.
5. **Implementation requirements** вЂ” the exhaustive catalog of 32 planted defects (V-01 through V-32), organized by module, each referencing an invariant where applicable.
6. **Machine-readable defect catalog** вЂ” a YAML block delimited by `<!-- BUGS-CATALOG-START -->` / `<!-- BUGS-CATALOG-END -->` that the scorer parses to extract ground truth.

**Axis distribution of the 32 defects:**

| Axis              | Count | Difficulty mix |
|-------------------|:-----:|----------------|
| correction        | 11    | 3 trivial, 4 medium, 4 hard |
| best-practices    | 5     | 2 trivial, 3 medium |
| utility           | 5     | 3 trivial, 2 medium |
| duplication       | 4     | 1 trivial, 3 medium |
| overengineering   | 2     | 1 trivial, 1 medium |
| documentation     | 2     | 1 trivial, 1 medium |
| tests             | 1     | 1 trivial |

Intentionally weighted toward `correction` (the most domain-testable axis), with each axis receiving at least one instance across multiple difficulty levels.

**Clean files:** three files are explicitly marked as defect-free (`src/index.ts`, `src/symbols/symbol.ts`, `src/paylines/standardPaylines.ts`). The scorer uses these to detect false positives in zones where Anatoly should find nothing.

### 2.7. Workflow

Fixture generation (infrequent вЂ” only when SPEC changes):

```bash
cd catalog/slot-engine

# Evolve reads ../SPEC.md as spec; project/ is the working tree.
# project/{package.json, tsconfig.json, README.md} are seeded and must not be modified.
evolve start project --spec ../SPEC.md --check ../verify.sh --rounds 30 --model claude-opus-4-6

# commit the generated project (stable until next spec change)
git add project
git commit -m "fixture(slot-engine): regenerate"
```

Bench run (frequent вЂ” every PR on Anatoly, or nightly):

```bash
# Anatoly audits the project; SPEC.md is physically absent from project/
anatoly run --target catalog/slot-engine/project

# Bench scores the report against the SPEC's YAML catalog
bench score \
  --report catalog/slot-engine/project/.anatoly/report \
  --spec catalog/slot-engine/SPEC.md \
  --baseline baselines/2026-04-23.json
```

At no point does Anatoly have access to `SPEC.md`, `verify.sh`, `verify-runtime.mts`, or any file containing the defect catalog вЂ” they all live in `catalog/slot-engine/`, one level above the directory Anatoly audits. The only README Anatoly sees is `project/README.md`, the neutral user-facing version that describes the library honestly and contains the DOC-RENAMED-API defect (a stale reference to `simulate()`).

### 2.8. `verify.sh` вЂ” how Evolve knows it has converged

Two-part check:

1. **Compilation**: `npx tsc --noEmit` exits 0.
2. **Empirical invariant violation**: a 100 000-spin Monte Carlo simulation measures the engine's RTP. The check **requires RTP > 1.00** (above 100%). This proves empirically that the business-invariant defects are present вЂ” if Evolve "fixes" a major defect (uniform reels, unbounded free spins, wild-substitutes-jackpot, etc.), the RTP drops below 100% and the check fails.

**Why this matters:** this is the strongest defense against Evolve's natural tendency to "improve" the code. A simple grep-based check can be gamed (Evolve could pass the grep while silently refactoring the surrounding logic). A Monte Carlo check on RTP is an emergent property вЂ” Evolve can only satisfy it by keeping the engine economically broken.

Additional targeted greps validate a few technical defects (e.g., `grep "Math.random()" src/rng/rng.ts`) to ensure the specific signatures expected by the scorer are present.

### 2.9. The scorer

Takes Anatoly's sharded report + the SPEC's YAML catalog. For each axis:

- **Matching**: bipartite matching between Anatoly's findings and the catalog's violations, keyed by `(file, line В± tolerance, axis, expected_verdict)`.
- **Metrics**: `recall = TP / (TP + FN)`, `precision = TP / (TP + FP)`, `F1`. Weighted-by-difficulty variant also computed (`trivial=1, medium=2, hard=3`).
- **Special cases**:
  - `utility` findings are keyed by `(file, symbol_name)` instead of `(file, line)`.
  - `duplication` findings describe **groups** of symbols; matching uses a set-based similarity (Jaccard в‰Ґ 0.5).
  - Project-wide defects (V-29 zero-test-coverage, V-30 no-JSDoc, V-32 config-mutation) have custom matching logic.
- **Output**:
  - JSON for CI (`{ global: 0.83, per_axis: {...}, misses: [...], fps: [...], cost_usd: 1.42, duration_s: 252 }`)
  - Markdown for humans (per-axis table, top misses with code context, false positives with Anatoly's reason, delta vs. baseline).

A **score of 1.0 requires an exact 1-for-1 match** between Anatoly's findings and the catalog вЂ” every planted defect detected, zero false positives. This is mathematically achievable but practically very hard, which is the intended property: the score should move when Anatoly improves, not plateau at 0.95 on day one.

## 3. What this design explicitly rejects

Choices that were considered and discarded, with reasons:

- **Inline annotations (`// @anatoly-expect вЂ¦`) in fixtures** вЂ” rejected because the stripper-before-audit mechanism is a fragile dependency. Physical separation (SPEC.md outside `project/`) is simpler and safer.
- **One fixture per defect (~70 mini-projects)** вЂ” rejected because tiny isolated fixtures cannot carry realistic noise, and thus cannot measure precision. A single realistic multi-axis project is more representative and cheaper to audit (7 runs в†’ 1 run).
- **Focused projects per axis (7 projects)** вЂ” initially proposed as a compromise, rejected in favor of fully mixed for the same reason: real users run Anatoly on mixed codebases, not on axis-isolated ones.
- **Grep-only verify.sh** вЂ” rejected in favor of Monte Carlo RTP check, because grep can be gamed by cosmetic refactors while RTP is an emergent property of the code's actual behavior.
- **Evolve generates the entire noise + defects** вЂ” accepted with caveats. We acknowledge this is the riskiest part of the design and plan for the possibility that Evolve may require multiple rounds, explicit anti-improvement language, or manual intervention on individual defects.
- **Multi-language coverage at launch** вЂ” deferred. TypeScript only for the first version.

## 4. Open questions (unresolved at end of discussion)

1. **Will Evolve cooperate with a spec that mandates preserving defects?** Untested. The primary risk. Mitigation: start with the slot-engine fixture, observe convergence behavior, adjust framing if Opus persistently "fixes" defects. Fallback: hand-write the most stubborn defects.
2. **How strict is the matching between Anatoly's `reason` text and the expected `nature`?** Current design matches only on `(file, line, axis, verdict)`. Semantic matching on the reason field (via embeddings or an LLM judge) is a possible v2 enhancement.
3. **Variance of Anatoly's scores across runs.** Undecided whether the bench's "official" number is a single-run score (temperature low) or an average of N=3вЂ“5 runs. First implementation: single run. Stability measurement as a separate, less frequent job.
4. **CI integration frequency.** Every PR on Anatoly vs. nightly vs. pre-release only. Depends on the actual cost of a bench run, to be measured after the first full execution.
5. **Baselines and regression thresholds.** When to "fail" a PR on bench regression. Open, probably handled as a separate tuning task once we have several data points.

## 5. Ethical and legal notes

- The slot-engine fixture is a **pure simulation**. No UI, no real-money interface, no gambling product. It exists exclusively as an evaluation artifact.
- The fixture intentionally includes code patterns that would be illegal in regulated gambling markets (e.g., V-12 session-based luck adjustment). These are present **because** they violate business invariants and test Anatoly's ability to detect them. They are not presented as acceptable implementations.
- Dark patterns documented in the real gambling industry (near-miss psychology, loss-disguised-as-win audio, force-loss after jackpot) are **deliberately excluded**. They are not needed to make the fixture rich, and we have no interest in providing a reference implementation for them.
- The seeded `project/README.md` describes the library as a transparent simulator with a declared RTP target. It does not hide the engine's nature.
- If the project is ever published or referenced publicly, the fixture's role as an audit-tool eval should be made explicit. It is not a library anyone should use to build a real slot machine.

## 6. Next concrete steps

In order:

1. **Scaffold the `anatoly-bench` repo** with the directory structure above (`src/` empty, `catalog/slot-engine/` with the three seed files).
2. **Finalize `SPEC.md`** вЂ” the draft produced in this discussion is the starting point. Review the 32 defect entries, adjust axis distribution if needed, confirm V-12 framing.
3. **Seed `project/README.md`** вЂ” short, neutral, functional description of the slot-engine library. No mention of defects. Referenced by Anatoly at audit time; intentionally carries the DOC-RENAMED-API stale reference to `simulate()`.
4. **Write `verify.sh`** вЂ” compile check + Monte Carlo RTP > 1.0 + targeted greps for signature defects.
5. **Seed `package.json` and `tsconfig.json`** вЂ” dependencies (none heavy; just `typescript` and the Node types), strict TS config.
6. **First Evolve run on the slot-engine fixture.** Observe: does Opus respect the "do not fix" framing? How many rounds to converge? Are any defects missed by Evolve (requiring manual patching)?
7. **First Anatoly audit** on the converged fixture. Generate a baseline report and manually inspect findings vs. catalog to sanity-check the approach before coding the scorer.
8. **Implement the scorer** (`src/score.ts`, `src/parser.ts`, `src/cli.ts`). Unit-test with synthetic findings against the YAML catalog.
9. **First official bench run**: Anatoly report в†’ scorer в†’ F1 score per axis and global. Capture as `baselines/YYYY-MM-DD.json`.
10. Decide based on results: is one reference project enough, or do we add a second? Is Evolve reliable enough, or do we switch to hand-written + Evolve-for-refactors?

## Appendix A вЂ” The seven axes of Anatoly (for context)

| Axis              | Default model | Verdicts                          | Detects                                                      |
|-------------------|---------------|-----------------------------------|--------------------------------------------------------------|
| utility           | Gemini Flash  | `USED` `DEAD` `LOW_VALUE`         | Dead code, unused exports, low-value code                    |
| duplication       | Gemini Flash  | `UNIQUE` `DUPLICATE`              | Semantically similar functions across the codebase           |
| correction        | Sonnet        | `OK` `NEEDS_FIX` `ERROR`          | Bugs, logic errors, API misuse (two-pass dependency check)   |
| overengineering   | Gemini Flash  | `LEAN` `OVER` `ACCEPTABLE`        | Excessive complexity relative to purpose                     |
| tests             | Sonnet        | `GOOD` `WEAK` `NONE`              | Per-symbol test coverage quality                             |
| best-practices    | Sonnet        | Score 0вЂ“10, 17 rules              | Language-specific best-practice violations (context-aware)   |
| documentation     | Sonnet        | `DOCUMENTED` `PARTIAL` `UNDOCUMENTED` | JSDoc gaps on exports, /docs/ desynchronization          |

Each axis runs in parallel per file, crash-isolated. The bench matches findings per axis and aggregates into a single score.
