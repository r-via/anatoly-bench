## What this fixture is for

`task-vault` is the **dead-code / dead-intent** fixture and the test bed for the
**`flow` axis** (doc 23 section 7). Where slot-engine ships numeric and
structural defects and train-dispatch ships behavioural-invariant defects,
task-vault ships two scored families:

1. **`utility`** (dead exports): the classic whole-symbol dead-code detector,
   plus the public-barrel false-positive guard.
2. **`flow`** (parcours-level structural quality): defects visible only across
   the cross-file flow graph, never to a per-symbol pass. Five natures:
   `unwired_flow`, `redundant_flow`, `inconsistent_siblings`, `over_indirect`,
   `effect_asymmetry`.

The composition is **mixed on purpose**. The flow graph and the dead-export pass
share an underlying fact (reachability from the public surface), so six symbols
carry **both** a `utility` DEAD finding and a `flow` `unwired_flow` finding at the
same `(file, symbol)`: a dead orphan that the flow skeleton also shows as an
unwired entry of a coherent parcours. This is the doc 22 section 5 / doc 23
section 5 "rich orphan" (dead-intent) synergy. The fixture measures it directly:
a detector that sees dead code but not dead intent (or the reverse) scores one
axis and misses the other on the same symbol.

It also exercises the deterministic, LLM-free `flows` phase (doc 23, Stage 1),
which writes `.anatoly/state/internal-flows/flows.json`. The `flow` axis consumes
that artifact. The phase output can also be inspected directly to debug the
barrel-aware walk (see "Inspecting the flows artifact" below).

## Fixture parameters

- **Project path**: `catalog/task-vault/project` (substitute for `<fixture>` in
  the shared doc's commands -> `task-vault`).
- **Scored axes**: 2, `utility` and `flow`. Every other axis is excluded via
  `scored_axes` in the catalog and disabled in `.anatoly.yml`. The project ships
  no tests and no JSDoc, and the live modules are deliberately clean, so any
  finding outside the cataloged cases on a scored axis is a precision error.
  `global_f1` is the bipartite-matched F1 across both scored axes;
  `per_axis.utility.f1` and `per_axis.flow.f1` are reported separately.
- **Ground truth**: [SPEC.md](./SPEC.md) (YAML catalog, never enters
  `project/`, never seen by Anatoly). 29 cataloged cases:
  - **`utility`, 10**: DEAD-ASSIGN (medium, an orphaned exported handler with no
    out-edge), DEAD-BACKLOG, DEAD-TEMPLATE, DEAD-LEGACY-TYPE (trivial), and the
    six dead orphans DEAD-PREVIEW / DEAD-TOUCH / DEAD-ARCHIVE-SUMMARY / DEAD-PING /
    DEAD-DIGEST / DEAD-RECOUNT (medium).
  - **`flow`, 19**: 6 `unwired_flow` (the six dead orphans again, dual-anchored),
    4 `redundant_flow` (noteTask/annotateTask, renameProject/retitleProject),
    4 `inconsistent_siblings` (importTask, forceComplete, rushArchive, seedUser),
    3 `over_indirect` (completeViaPipeline, assignViaRouter, announceProject),
    2 `effect_asymmetry` (registerUser, notifyAssignee).
- **Determinism**: no RNG and no clock dependence (ids derive from the current
  count). A single `verify-runtime.mts` run drives the public API and is the
  runtime ground truth; dead-export orphan status and the flow shapes are checked
  by static greps and function-body extraction in `verify.sh`.

## Validate the fixture BEFORE benching (Evolve convergence)

This fixture is authored the slot-engine way: the `src/` tree is generated by
Evolve to converge against `SPEC.md`, not hand-written. The files
`project/package.json`, `project/tsconfig.json`, and `project/README.md` are
seeded and must stay fixed; Evolve generates the `src/` tree. Before any scored
run:

```bash
# on the VM, inside the project dir
cd ~/project/anatoly-bench/catalog/task-vault/project
evolve   # reads evolve.toml: check=../verify.sh, spec=../SPEC.md
```

Convergence means `verify.sh` exits 0:

- the code typechecks,
- the public barrel re-exports exactly the public surface (and no dead export
  leaks into it),
- exactly the 12-module layout is present,
- the live mutators, effects, and pure helpers are wired in (referenced by at
  least two modules), so a correct audit never flags them dead,
- each of the 10 cataloged dead exports is present in source yet referenced by no
  other module,
- each defective flow keeps its exact structural shape: the six unwired orphans
  reach no effect; the redundant twins both reach the shared effect; the
  inconsistent siblings reach the effect but omit the named guard their reference
  sibling performs (`isBlank`, `readTask`, `readProject`, `readUser`); the
  over-indirect chains keep every private pass-through hop; the user store and the
  outbox stay asymmetric (no teardown mutator, no outbox drain),
- the public API behaves correctly when driven by `verify-runtime.mts`.

If a round wires in or deletes a dead export, harmonizes a defective flow with its
clean sibling, shortens an over-indirect chain, adds a missing guard, drains the
outbox, or breaks the public surface, convergence fails. A stray module or helper
fails the exact-12-modules check (a spurious orphan would corrupt precision).

After convergence, re-validate the catalog `(file, symbol)` anchors against the
generated tree (see the gotcha below) before trusting any baseline.

## Inspecting the flows artifact

`flows: { enabled: true }` in `.anatoly.yml` makes the run emit
`project/.anatoly/state/internal-flows/flows.json`. The `flow` axis scores from
this; inspect it to debug the barrel-aware walk when a `flow` verdict looks wrong.
For the converged tree expect, in line with SPEC.md "Flow-phase structural
expectations":

- entry flows rooted at every barrel-exported command (`addTask`, `completeTask`,
  `reopenTask`, `removeTaskCmd`, `createProject`, `archiveProject`,
  `registerUser`, ...), each `unwired: false` (each reaches a state-mutation or
  outbox effect),
- entry flows rooted at the six `async` dead orphans (`previewTask`, `touchTask`,
  `summarizeArchive`, `pingUser`, `buildDigest`, `recountOpen`), each
  `unwired: true` (they resolve out-edges to pure reads/formats but reach no
  effect): the dead-intent candidates,
- entry flows rooted at `loadSummary` and `loadProjectReport`, also `unwired: true`
  (read-only) but public surface: the `flow` axis must treat them as legitimate
  queries, NOT as dead intent (the unwired false-positive guard),
- no entry flow for the internal mutators (`insertTask`, `markTaskDone`, ...) or
  the pure helpers: each has a real internal consumer, so the graph root sees a
  genuine caller,
- no entry flow for `assignTask` or `computeBacklog`: each references no project
  function, so it has no resolvable out-edge and the walk emits nothing; they
  surface only on `utility`.

```bash
# on the VM, after a run
python3 -m json.tool \
  ~/project/anatoly-bench/catalog/task-vault/project/.anatoly/state/internal-flows/flows.json \
  | less
```

These expectations depend on the rag-index NLP summariser (Haiku) assigning the
intended `behavioralProfile` to each function (the mutating commands and
`notifyAssignee` / `announceProject` as `sideEffectful`, the store mutators as
`stateful`, the read queries and dead orphans as `async`, the pure helpers as
pure). If an entry is missing or an `unwired` flag is wrong, check the assigned
profiles in the rag index before suspecting the walk.

## Scoring

After copying the run back (see shared doc), score on the local machine:

```bash
cd ~/projects/anatoly-bench
npm run build  # only if the bench source changed

VERSION=v1   # increment from the last baseline
DATE=$(date +%Y-%m-%d)
SPEC=catalog/task-vault/SPEC.md
RUN=catalog/task-vault/project/.anatoly/runs/$RUN_ID

node dist/cli.js score \
  --spec $SPEC \
  --report $RUN \
  --json baselines/${DATE}_task-vault-${VERSION}.json \
  --md   baselines/${DATE}_task-vault-${VERSION}.md
```

The JSON score contains `global_f1`, `per_axis`, `misses` and `false_positives`;
the Markdown also surfaces run metadata (commit, duration, cost, tokens). With two
scored axes, read `per_axis.utility.f1` and `per_axis.flow.f1` separately:
`global_f1` is their bipartite-matched aggregate, so a strong utility score can
mask a weak flow score and vice versa.

## task-vault-specific gotchas

- **The barrel guard is the point (utility).** The entire public surface is
  imported by no module except `src/index.ts` (a re-export). If the detector lists
  any public command or query in `false_positives`, its reachability is not seeing
  through public re-export barrels: the exact regression this fixture guards. Watch
  `false_positives` as closely as `misses`.
- **Six symbols are dual-anchored.** previewTask, touchTask, summarizeArchive,
  pingUser, buildDigest, and recountOpen each carry a `utility` DEAD finding AND a
  `flow` `unwired_flow` finding at the same `(file, symbol)`. The scorer matches
  per axis, so the two are counted independently: a run can score the utility
  finding and miss the flow one (dead code seen, dead intent not), or the reverse.
  Read both axes' `misses` on these symbols.
- **The flow false-positive guards are load-bearing.** `loadSummary` and
  `loadProjectReport` are structurally unwired (read-only) but public, so they must
  NOT be flagged `unwired_flow`. The task and project stores are symmetric
  (`removeTaskCmd`, `archiveProject`), so they must NOT be flagged
  `effect_asymmetry`. The well-formed sibling of each inconsistent pair (`addTask`,
  `completeTask`, `archiveProject`, `registerUser`) and the correct commands that
  reach no twinned effect must NOT be flagged. A `flow` finding on any of these is
  a precision error.
- **`inconsistent_siblings` is asymmetric.** Only the deficient sibling
  (`importTask`, `forceComplete`, `rushArchive`, `seedUser`) is cataloged; the
  reference sibling is not. A detector that flags both, or flags the wrong one,
  scores a false positive plus a miss.
- **Both redundant twins are cataloged.** The redundancy implicates both flows, so
  noteTask AND annotateTask (and renameProject AND retitleProject) are expected
  findings. Flagging only one is a miss.
- **Three dead exports share a file each.** legacy.ts holds DEAD-BACKLOG
  (`computeBacklog`) and DEAD-LEGACY-TYPE (`LegacyVault`); tasks.ts holds several.
  The scorer matches on `(file, symbol)`; a file-level-only finding matches at most
  one. Watch `misses` to tell which the detector saw.
- **DEAD-ASSIGN is the hard utility case.** `assignTask` is a wide-span, plausible,
  exported command handler whose siblings are live, so the module is not dead, only
  this one export. It references no project function (no out-edge), so it never
  appears on `flow`: a flow finding on `assignTask` is a false positive.
- **The flow axis is young.** Of the five natures, `unwired_flow` is the most
  directly computable from the current walk; the cross-flow natures
  (`redundant_flow`, `inconsistent_siblings`, `over_indirect`, `effect_asymmetry`)
  depend on detectors that land incrementally. Expect `per_axis.flow.f1` to start
  low and climb as those detectors mature. This fixture is the regression and
  progress tracker for that build-out.
- **Stable `project/` tree.** If `project/` is regenerated by Evolve, re-validate
  the cataloged `(file, symbol)` anchors before trusting a baseline. The catalog
  anchors on `file` + `symbol` rather than line hints, which is robust to
  regeneration.
